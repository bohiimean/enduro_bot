# Telegram-бот — техническое задание

Статус: архитектура и стек зафиксированы, разработка не начата.
Этот документ — источник истины по принятым решениям. Не пересматривать архитектуру без явного запроса.

## Контекст

Низкая нагрузка — ориентир ~10 пользователей в час. Все архитектурные решения исходят из этого масштаба. БД, Redis, Celery, Django сознательно не используются — они избыточны при такой нагрузке и добавляют сложность без выгоды. Если нагрузка вырастет до 400-500+ чел/час — см. раздел "Порог пересмотра архитектуры" в конце файла.

Telegram заблокирован в РФ — бот обязан работать на зарубежном VPS, российский хостинг не вариант (сервер физически не достучится до api.telegram.org).

## Функционал

### 1. Курс валют

Пользователь нажимает кнопку — бот присылает одно сообщение с курсом и итоговой ценой по каждому способу оплаты (курс + наценка).

**USDT/RUB** — решено:
- Источник: Rapira API, публичный эндпоинт, без авторизации
  ```
  GET https://api.rapira.net/open/market/rates
  ```
  Лимит: 5 запросов/сек, не более 100/мин — для нашей нагрузки избыточный запас.
  В ответе ищем пару `USDT/RUB`, берём `close` (или среднее `bidPrice`/`askPrice`).
- Итоговая цена = курс × 1.045 (комиссия/наценка +4.5%, фиксированная).
- Доступные на Rapira пары (для справки, остальные сейчас не используются): `USDT/RUB`, `BTC/USDT`, `TRX/USDT`, `ETH/USDT`, `ETH/BTC`, `USDC/USDT`, `BNB/USDT`. Пары USD/RUB на Rapira нет — это крипто-биржа.

**USD/RUB** — НЕ РЕШЕНО, отложено клиентом. Пока заглушка (см. раздел "Реализация курса" — `UsdRubStubProvider`). Рассмотренные и отклонённые/возможные варианты — см. "Открытые вопросы".

**Кеширование курса**: in-memory, TTL 3-5 минут (точное значение — конфигурируемо). Не дёргать внешний API на каждое сообщение пользователя.

### 2. Статус заказа

Пользователь вводит номер телефона — бот возвращает статус заказа(ов). Клиент подтвердил: поиск по номеру телефона (`ФИО` в таблице — для отображения/проверки совпадения, не ключ поиска). В реальной таблице нет отдельного номера заказа, поэтому это единственный рабочий вариант, не опция на выбор.

Источник данных — Google Таблица, лист `Orders`. **Прямых обращений к Google Sheets на каждый запрос пользователя нет** — данные читаются в память при старте и обновляются по таймеру (см. "Реализация Google Sheets").

Клиент продолжает вести таблицу как обычно — никакой админки, никакой миграции в БД на этом этапе.

### 3. Каталог товаров (наличие)

Категории (по бренду, колонка `Марка`) → список моделей с пагинацией → карточка (модель, цвет, цена, наличие, кнопка «Связаться с менеджером»).

Покупки через бота не делаем — только просмотр и контакт с менеджером (решено).

Источник данных — тот же лист `Orders`, не отдельный лист. Каталог — это строки Orders, где колонка `Статус` = "в наличии" (см. "Реализация Google Sheets").

Навигация по каталогу — через `callback_data` кнопок (весь контекст: категория, страница, id товара кодируется прямо в кнопке), без хранения состояния пользователя на сервере:
```
cat:{category}:page:{n}
item:{product_id}
```

## Технический стек

| Компонент | Технология | Комментарий |
|---|---|---|
| Бот | aiogram 3.x (Python, async) | long polling, не webhook |
| Данные (заказы, товары) | Google Sheets API | сервисный аккаунт, read-only |
| Кеш курса и листов | in-memory (Python dict + TTL) | без Redis |
| Фоновые задачи | APScheduler | без Celery |
| БД | нет | при текущей нагрузке не нужна |
| Контейнеризация | Docker + docker-compose | один контейнер, без оркестрации (k8s избыточен) |
| Хостинг | зарубежный VPS | обязательно — Telegram заблокирован в РФ |

**Почему long polling, а не webhook**: webhook требует HTTPS-эндпоинт, сертификат, nginx. При такой нагрузке это чистые накладные расходы без выгоды. Polling — один Python-процесс, без веб-сервера.

**Почему без БД**: единственный аргумент за БД на этом масштабе — защита от случайной потери данных в таблице (если клиент удалит строку). Это не техническая необходимость, а вопрос договорённости с клиентом. Сейчас не реализуем.

## Реализация курса валют

Интерфейс провайдера курса — чтобы безболезненно подключить реальный источник USD/RUB, когда он определится:

```python
class RateProvider:
    async def get_rate(self) -> Decimal: ...

class RapiraProvider(RateProvider):
    MARKUP = Decimal("1.045")  # +4.5%, решено и зафиксировано

    async def get_rate(self) -> Decimal:
        data = await fetch("https://api.rapira.net/open/market/rates")
        usdt_rub = next(p for p in data["data"] if p["symbol"] == "USDT/RUB")
        return Decimal(str(usdt_rub["close"])) * self.MARKUP

class UsdRubStubProvider(RateProvider):
    """Источник USD/RUB не определён. Заглушка — заменить, когда решится."""
    async def get_rate(self) -> Decimal:
        return Decimal("0")
```

Курс достаётся через кеш с TTL и `asyncio.Lock`, чтобы при параллельных запросах не было повторных обращений к API:

```python
class RateCache:
    def __init__(self, ttl_seconds: int):
        self._ttl = ttl_seconds
        self._store: dict[str, tuple[Decimal, float]] = {}
        self._lock = asyncio.Lock()

    async def get_or_refresh(self, key: str, provider: RateProvider) -> Decimal:
        async with self._lock:
            cached = self._store.get(key)
            if cached and time.time() - cached[1] < self._ttl:
                return cached[0]
            rate = await provider.get_rate()
            self._store[key] = (rate, time.time())
            return rate
```

**Фолбэк при сбое Rapira API**: вернуть последнее закешированное значение с пометкой времени ("курс на HH:MM"), не ронять хендлер. Залогировать ошибку.

## Реализация Google Sheets

Схема подтверждена реальным фрагментом таблицы (`фрагмент.xlsx`) — клиент сообщил, что структура 1:1 совпадает с Google-таблицей.

**Orders** — единственный лист, используется и для статуса заказа, и для каталога (наличия). Отдельного листа под каталог нет — это уточнено клиентом явно.

| дата | Цена | Оплачено | Остаток | Марка | Модель | Прог-я | Цвет | кол-во | Статус поставки | Статус | ФИО | Телефон | Город | ТК |
|---|---|---|---|---|---|---|---|---|---|---|---|---|---|---|

`Статус` — новая колонка (добавляется клиентом), значения `"в наличии"` / `"нет в наличии"`. Не путать с `Статус поставки` (свободный текст про ETA доставки) — это две разные колонки, существуют одновременно.

Важные особенности реальных данных, влияющие на реализацию:

- **Номера заказа как отдельной колонки нет.** Уникального ID в таблице не существует. Поиск возможен только по `Телефон` — клиент подтвердил это явно.
- **Один телефон — возможно несколько строк.** Один клиент может иметь несколько заказов (разные модели/даты). Бот обязан возвращать все совпадения, а не первое найденное.
- **"Статус поставки" — свободный текст, не enum.** Реальные значения: `"в пути - начало июля в Москве"`, `"после 20 июня в Москве"`, `"свяжитесь со мной по этой модели в понедельник"`. Оператор вписывает текст руками, фиксированного списка статусов нет. Бот отдаёт это значение как есть — никакой нормализации или маппинга на enum.
- **Каталог (п.3 ТЗ) = строки этого же листа, где `Статус` = "в наличии".** Решено клиентом: фильтр оставить простым, без разделения "общий сток"/"персональный заказ" — это известное ограничение, доработается по факту использования. При выводе карточки в каталоге **обязательно скрывать** `ФИО`, `Телефон`, `Город`, `ТК` — показывать только `Марка`/`Модель`/`Цвет`/`Цена`. Это не опция, а требование — иначе в публичный каталог утекут персональные данные чужого заказа.
- **`Прог-я` (link / no link)** — назначение не до конца понятно (возможно, отметка о наличии трек-номера у партии). **Уточнить у клиента**, нужно ли показывать это поле пользователю или оно служебное, для внутреннего использования.
- `дата` хранится как число (серийная дата Excel/Sheets) — при выводе клиенту форматировать в `ДД.MM.ГГГГ`.

---

Доступ к таблице — через сервисный аккаунт Google Cloud (Sheets API включить в проекте, JSON-ключ на сервере, таблица расшарена на email сервисного аккаунта с правом просмотра — запись не нужна, бот только читает).

Кеш листа в памяти, обновление по таймеру через APScheduler (не на каждый запрос пользователя):

```python
class SheetsCache:
    def __init__(self, spreadsheet_id: str):
        self.spreadsheet_id = spreadsheet_id
        self._orders: list[dict] = []

    async def refresh(self):
        self._orders = await fetch_sheet(self.spreadsheet_id, "Orders")

    def find_orders_by_phone(self, phone: str) -> list[dict]:
        normalized = normalize_phone(phone)
        return [o for o in self._orders if normalize_phone(o["Телефон"]) == normalized]

    def available_items(self) -> list[dict]:
        # Решено клиентом: без доп. фильтра по Телефон, оставить как есть.
        # Известное ограничение — доработается по факту использования.
        return [o for o in self._orders if o["Статус"] == "в наличии"]
```

**Важно для хендлера каталога**: при формировании карточки товара использовать только `Марка`/`Модель`/`Цвет`/`Цена` из результата `available_items()`. Поля `ФИО`, `Телефон`, `Город`, `ТК` в каталог не передавать и не выводить — это персональные данные конкретного заказа, а не атрибуты товара. Сделать это явным шагом маппинга (отдельная функция `to_catalog_card(item: dict) -> dict`), а не полагаться на то, что хендлер "просто не возьмёт" лишние поля — явный allowlist полей безопаснее, чем подразумеваемый.

`normalize_phone()` — обязательно нормализовать номер перед сравнением (убрать пробелы, скобки, дефисы, привести `8`/`+7` к одному виду), иначе совпадения будут теряться из-за разного формата ввода у клиентов.

Интервал обновления — 5-10 минут (конфигурируемо, см. `.env`).

**Фолбэк при сбое обновления**: если плановый рефреш не удался — оставить предыдущие данные в кеше, не перезатирать пустым результатом. Залогировать ошибку.

## Деплой

- Зарубежный VPS (Hetzner, DigitalOcean и т.п.) — обязательное условие, не опция.
- **Запуск через Docker** (решено добавить контейнеризацию — без неё дальше не двигаемся).

```dockerfile
# Dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["python", "main.py"]
```

```yaml
# docker-compose.yml
services:
  bot:
    build: .
    restart: unless-stopped
    env_file: .env
    volumes:
      - ./credentials.json:/app/credentials.json:ro
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
```

- JSON-ключ сервисного аккаунта Google монтируется volume'ом, не запекается в образ.
- Логи — стандартный `logging` в stdout, Docker сам пишет их через `json-file` driver (лимиты выше не дают логам разрастаться бесконечно — отдельный logrotate не нужен).
- Перезапуск при падении — `restart: unless-stopped`. Отдельный systemd-юнит не нужен; если на сервере уже настроен автозапуск Docker при перезагрузке — `docker compose up -d` достаточно.
- Webhook/nginx/SSL всё ещё не нужны — long polling работает из контейнера так же, как из обычного процесса.

## Отклонённые подходы (не реализовывать, если не попросят явно)

- **Скрапинг investing.com напрямую** — сайт активно защищается от ботов (Cloudflare), нет официального API, риск нарушения ToS.
- **CyberScraper 2077 / AI-скраперы для investing.com** — отклонено. Причины: (1) капча в инструменте решается вручную через GUI-браузер — несовместимо с headless-сервером без монитора; (2) LLM-вызов на каждое извлечение одного числа — абсурдно дорого и медленно относительно задачи; (3) Cloudflare-bypass — гонка вооружений, нестабильно в долгосрочной перспективе; (4) лишняя нагрузка на ресурсы VPS (полноценный браузерный движок Playwright).
- **MOEX как источник USD/RUB** — отклонено. Биржевые торги USD/EUR на Московской бирже приостановлены с июня 2024 (санкции против самой биржи и НКЦ), официальный курс ЦБ сейчас формируется не по биржевым данным.
- **Excel вместо Google Таблиц** — отклонено, остаёмся на Google Sheets.
- **Django + PostgreSQL + Redis + Celery** — отклонено для текущего масштаба. Полная админка на Django была первоначальным планом, но избыточна при ~10 чел/час. Зафиксировать как вариант для будущего роста (см. ниже).

- **Каталог (`available_items`) — фильтровать ли дополнительно по пустому `Телефон`?** Решено клиентом: пока не фильтровать, оставить как есть (`Статус` = "в наличии"). - **Назначение колонки `Прог-я`** (link/no link) в листе Orders — нужно уточнить у клиента, показывать ли пользователю.

## Порог пересмотра архитектуры

Если нагрузка вырастет до **400-500+ пользователей в час** — текущая архитектура (Google Sheets + in-memory кеш) перестаёт быть надёжной (риск рейт-лимитов и потери запросов при пиках). В этом случае переходить на:
- PostgreSQL вместо прямого чтения Google Sheets
- Django + Django Admin для управления заказами/товарами
- Redis вместо in-memory кеша
- Celery + Celery Beat вместо APScheduler
- Скрипт единоразовой миграции данных из Google Sheets в БД

До этого порога — не реализовывать, это другой проект.
